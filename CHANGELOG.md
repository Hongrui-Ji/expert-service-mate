## 2026-03-29（排班能力扩展：月度目标、临时上门、门店停用）

### 功能变更
- 排班日历默认视图改为周视图；登录/切换账号后默认展示当前账号主体专家的排班视图
- 管理员支持“门店月度目标次数”覆盖（可增减、可为 0），并影响当月待排门店与进度计算
- 专家支持自建“临时上门”（报修/复查等），默认不计入常规目标次数；管理员可选择计入
- 临时上门在日历中用橙色样式区分，并展示原因（title）
- 历史月份不显示待排门店列表（仅用于查看），避免新增/恢复门店导致历史月份出现待排
- 登录状态增强：本地检测 Token 过期并提示重新登录；服务端将过期/无效 Token 统一返回 401
- 门店库增强：新增“特殊需求”筛选；特殊需求仅管理员可修改

### 门店删除策略调整（物理删除 → 停用/归档，默认策略 A）
- 物理删除门店改为“停用门店（软删除）”，保留历史排班与审计价值
- 默认策略 A：若门店存在今天及未来的 planned 排班，停用会被阻止（返回 409），需先取消/改期后再停用
- 支持“恢复门店”，恢复后重新参与排班
- 门店停用后禁止创建新的排班/临时上门（返回 409）

### 数据库结构变更
- `stores` 新增字段：
  - `deleted_at`：停用时间；非空表示门店已停用/归档
  - `service_start_month`：门店从该月开始纳入常规待排计算（新增门店默认=创建当月）
  - `service_resume_month`：门店恢复后从该月开始重新纳入常规待排计算（恢复门店默认=恢复当月）
- 新增表：
  - `store_month_plans`：门店月度目标次数覆盖
- `visits` 扩展字段：
  - `type`：`regular`（常规）/ `extra`（临时）
  - `title`：临时上门原因（extra 必填）
  - `count_towards_target`：是否计入当月常规目标次数
  - `created_by`、`created_at`

### 接口变更
- 认证
  - `POST /api/auth/login` 签发 Token 有效期调整为 7 天
- 月度计划
  - `GET /api/store-month-plans?month=YYYY-MM`
  - `PUT /api/stores/:id/month-plan`（Admin）
  - `DELETE /api/stores/:id/month-plan?month=YYYY-MM`（Admin）
- 门店停用/恢复
  - `DELETE /api/stores/:id`：语义改为停用门店（Admin，策略 A）
  - `POST /api/stores/:id/restore`：恢复门店（Admin）
- 排班/临时上门
  - `POST /api/visits` 支持 `type/title/countTowardsTarget`
  - 权限：普通用户仅允许创建/更新/删除 `expert_name=本人` 的 visits

### 影响范围
- 前端排班页：待排门店计算、日历卡片样式、临时上门创建入口
- 前端门店管理页：新增“本月计划”入口；删除按钮语义变为停用并支持恢复
- 后端：外键约束强制开启（`foreign_keys=ON`）；新增/扩展表与接口
- 文档：已同步更新架构文档的 Schema 与 API 权限矩阵

### 验收要点（推荐顺序）
- 账号切换后默认展示本人日历视图；管理员可切换查看其他专家视图
- 管理员对某门店设置本月目标次数后，待排列表与进度按覆盖值变化（含目标为 0）
- 创建临时上门后：
  - 日历中显示为橙色卡片/标签
  - 默认不计入当月常规目标次数
- 停用策略 A：
  - 门店存在未来 planned 时，停用被阻止并提示原因
  - 取消/改期后可停用；停用后不可新建排班/临时上门；可恢复
